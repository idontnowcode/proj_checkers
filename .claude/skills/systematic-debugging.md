# Systematic Debugging

버그나 예상치 못한 동작을 마주쳤을 때, 무작위 수정 없이 근본 원인을 찾아 해결하는 방법론.

> **플랫폼 중립**: 이 스킬은 도구 호출이 없는 순수 방법론이다. Claude Code, Codex 모두 동일하게 적용된다.

---

## 철칙

```
근본 원인 파악 전에는 어떤 수정도 시도하지 않는다.
```

증상만 고치는 것은 실패다.

---

## 4단계 프로세스

각 단계를 완료하기 전에 다음 단계로 진행하지 않는다.

### Phase 1: 근본 원인 조사

**수정 시도 전에 반드시:**

1. **오류 메시지 정독**
   - 오류와 경고를 건너뛰지 않는다
   - 스택 트레이스 전체를 읽는다
   - 줄 번호, 파일 경로, 오류 코드를 기록한다

2. **재현 조건 확인**
   - 항상 발생하는가, 특정 조건에서만 발생하는가?
   - 정확한 재현 단계가 무엇인가?
   - 재현 불가 시 → 추측 금지, 데이터 수집 후 재시도

3. **최근 변경 사항 확인**
   - 무엇이 바뀌었길래 이 문제가 생겼는가?
   - git diff, 최근 커밋 확인
   - 새 의존성, 설정 변경, 환경 차이

4. **다중 컴포넌트 시스템의 증거 수집**

   시스템에 여러 레이어가 있다면 (A → B → C 흐름):
   ```
   각 컴포넌트 경계에서:
   - 입력되는 데이터 로깅
   - 출력되는 데이터 로깅
   - 어느 레이어에서 깨지는지 확인
   
   증거를 모은 뒤 → 실패 컴포넌트 특정 → 해당 부분만 조사
   ```

5. **데이터 흐름 추적**

   오류가 콜스택 깊숙이 있다면:
   - 잘못된 값이 어디서 시작됐는가?
   - 무엇이 이 값을 넘겼는가?
   - 계속 거슬러 올라가 원점을 찾는다
   - 증상이 아닌 원점에서 수정한다

---

### Phase 2: 패턴 분석

**수정 전에 패턴을 찾는다:**

1. **작동하는 유사 코드 찾기**
   - 같은 코드베이스에서 비슷하게 작동하는 부분이 있는가?
   - 작동하는 것과 안 되는 것의 차이가 무엇인가?

2. **차이점 목록화**
   - 아무리 사소해 보여도 모든 차이를 나열한다
   - "그게 문제일 리 없어"라고 가정하지 않는다

3. **의존성 파악**
   - 이 코드가 필요로 하는 다른 컴포넌트, 설정, 환경변수는?
   - 어떤 가정을 하고 있는가?

---

### Phase 3: 가설 및 테스트

**과학적 방법:**

1. **단일 가설 수립**
   - "X가 근본 원인이라고 생각한다. 왜냐하면 Y이기 때문이다."
   - 구체적으로 명시한다

2. **최소 변경으로 테스트**
   - 가설을 검증할 수 있는 가장 작은 변경만 한다
   - 한 번에 하나의 변수만 바꾼다
   - 여러 수정을 동시에 하지 않는다

3. **결과 확인 후 진행**
   - 작동하면 → Phase 4로
   - 작동 안 하면 → 새 가설 수립 (기존 수정에 추가하지 않는다)

4. **모르면 솔직하게**
   - "X를 이해하지 못하겠다"고 말한다
   - 아는 척하지 않는다

---

### Phase 4: 구현

**증상이 아닌 근본 원인을 수정한다:**

1. **재현 케이스 먼저 작성**
   - 가능하면 자동화된 테스트
   - 불가능하면 수동 재현 절차
   - 수정 전에 반드시 존재해야 한다

2. **단일 수정**
   - 파악된 근본 원인만 수정
   - "여기 온 김에" 다른 것도 고치지 않는다
   - 리팩터링 번들 금지

3. **수정 검증**
   - 해당 케이스가 통과하는가?
   - 다른 테스트가 깨지지 않는가?

4. **수정이 안 먹힌다면**
   - 멈춘다
   - 몇 번 시도했는지 센다
   - 2회 미만: Phase 1로 돌아가 새 정보로 재분석
   - **3회 이상: 아키텍처 문제일 가능성 높음 → 사용자와 논의**

---

## 🚩 이 생각이 들면 즉시 멈추고 Phase 1로

| 생각 | 현실 |
|------|------|
| "일단 이것부터 바꿔보자" | 원인도 모르면서 수정하는 것 |
| "아마 X가 문제일 거야" | 추측 기반 수정 = 시간 낭비 |
| "이건 단순한 문제야" | 단순한 버그도 근본 원인이 있다 |
| "급하니까 빠르게 해야 해" | 체계적 접근이 결국 더 빠르다 |
| "수정을 여러 개 한꺼번에 하면 빠를 것 같아" | 무엇이 효과 있었는지 알 수 없다 |
| "또 다른 수정 시도해보자" (3회+) | 이제는 아키텍처 문제를 의심할 때 |

---

## 빠른 참조

| 단계 | 핵심 활동 | 완료 기준 |
|------|----------|-----------|
| Phase 1 | 오류 읽기, 재현, 변경 확인, 증거 수집 | 무엇이 왜 깨졌는지 이해 |
| Phase 2 | 작동 예시 찾기, 차이 비교 | 차이점 목록 완성 |
| Phase 3 | 가설 수립, 최소 테스트 | 가설 확인 또는 새 가설 |
| Phase 4 | 테스트 작성, 수정, 검증 | 버그 해결, 테스트 통과 |
