# app-tester

test-planner가 생성한 Claude 테스트 체크리스트를 실제로 실행하여 구현 코드가 스킬 .md 명세대로 동작하는지 코드 분석으로 검증하고, 항목별 PASS/FAIL 리포트를 출력하는 스킬. 일반적인 코드 품질 평가가 아니라 **스킬 .md 명세 대조 검증**이 이 스킬의 핵심이다.

## 사용법

```
/app-tester
```

실행 후 다음 두 가지를 제공하면 된다:

1. **테스트 체크리스트** (필수): test-planner가 생성한 Claude 테스트 항목 목록
2. **구현 코드 파일 경로** (필수): 검증 대상 코드 파일들 (파일명, 디렉토리 경로 등)

두 가지를 한 번에 붙여넣어도 되고, 스킬이 순차적으로 요청할 수도 있다.

> **워크플로우 연계**
> - **자동 연계**: `test-planner` 완료 후 그 출력(체크리스트)을 받아 자동으로 실행된다.
> - **직접 호출**: `/app-tester`로 언제든지 독립 실행 가능하다.
> - **재작업 트리거**: FAIL 항목이 존재하면 `task-orchestrator`가 이 결과를 받아 최대 2회 자동 재작업을 수행한다.

## 동작 방식

### 1단계: 입력 수집 및 파싱

체크리스트와 구현 코드 파일 경로를 입력받는다.

- **체크리스트 파싱**: 각 체크 항목을 개별 검증 단위로 분리한다. 항목 번호, 검증 대상 동작, 관련 스킬 .md 명세 위치를 추출한다.
- **코드 파일 확인**: 제공된 경로의 파일들이 실제로 존재하는지 확인한다. 존재하지 않는 파일이 있으면 즉시 사용자에게 알린다.
- **명세 파일 로드**: 체크리스트가 참조하는 스킬 .md 파일을 다음 순서로 탐색한다.
  1. 체크리스트에 명세 파일 경로가 명시된 경우 → 해당 경로 직접 사용
  2. 체크리스트에 스킬 이름만 있는 경우 → `.claude/skills/<스킬명>.md` 경로 자동 시도
  3. 위 두 방법 모두 실패한 경우 → 사용자에게 명세 파일 경로를 직접 요청
  명세 파일 없이는 PASS/FAIL 판정 기준이 없으므로 반드시 확보한 후 다음 단계로 진행한다.

**이 단계의 산출물**: 파싱된 체크 항목 목록 + 로드된 코드 파일들 + 참조 명세 내용

### 2단계: 항목별 코드 분석 및 PASS/FAIL 판정

각 체크 항목에 대해 다음 절차로 판정한다.

**판정 기준: 스킬 .md 명세 대조**
일반적인 코드 품질(가독성, 성능, 패턴 등)이 아니라 스킬 .md 명세에 기술된 동작 요구사항이 구현 코드에 존재하는지가 유일한 기준이다.

**판정 절차**

| 순서 | 행동 |
|------|------|
| ① | 체크 항목이 참조하는 .md 명세 구절을 정확히 인용한다 |
| ② | 해당 명세를 구현하는 코드 위치(파일명:라인)를 탐색한다 |
| ③ | 코드가 명세의 요구사항을 실제로 충족하는지 판단한다 |
| ④ | PASS 또는 FAIL을 판정하고 근거를 기록한다 |

**PASS 조건**: 명세에 기술된 동작이 코드에 구현되어 있고, 그 구현이 명세의 의도를 벗어나지 않는다.

**FAIL 조건**: 아래 중 하나라도 해당하면 FAIL이다.
- 명세에 명시된 기능이 코드에 전혀 없다
- 코드가 존재하지만 명세와 반대되거나 상이한 동작을 한다
- 명세에서 필수(must/반드시)로 지정한 조건이 누락되었다

**이 단계의 산출물**: 항목별 판정 결과 (PASS/FAIL + 근거 + 코드 위치)

### 3단계: 리포트 생성 및 출력

2단계 결과를 아래 출력 형식에 따라 정리하여 출력한다. FAIL 항목이 하나라도 있으면 최종 판정은 "불합격"이다.

## 출력 형식

```
## app-tester 실행 결과

**검증 대상**: (스킬명 또는 파일명)
**체크 항목 수**: N개
**실행 일시**: YYYY-MM-DD

---

### 항목별 PASS/FAIL 결과표

| # | 체크 항목 | 판정 | 근거 |
|---|-----------|------|------|
| 1 | (항목 설명) | ✅ PASS | (코드 위치: 파일명:라인, 또는 "명세 충족 확인") |
| 2 | (항목 설명) | ❌ FAIL | (파일명:라인 — 명세와 불일치하는 이유) |
| 3 | (항목 설명) | ✅ PASS | (코드 위치) |
...

---

### FAIL 항목 상세

#### [#2] (항목 설명)
- **명세 요구사항**: "(스킬 .md에서 직접 인용한 해당 구절)"
- **코드 위치**: `파일명:라인번호`
- **원인**: (무엇이 구현되지 않았거나 잘못 구현되었는지 구체적으로)
- **권장 수정 방향**: (어떻게 고쳐야 명세를 충족하는지)

---

### 전체 합격 여부

| 항목 | 결과 |
|------|------|
| 전체 체크 항목 수 | N개 |
| PASS | N개 |
| FAIL | N개 |
| **최종 판정** | **✅ 합격 (FAIL 0건)** 또는 **❌ 불합격 (FAIL N건)** |

---

### 사용자 전달용 테스트 시나리오

> 아래는 사람이 직접 수행해야 하는 테스트 항목입니다. 코드 분석으로 검증할 수 없는 런타임 동작, UI/UX, 외부 연동 등을 포함합니다.

1. (시나리오 설명 — 어떤 입력으로, 어떤 동작을 확인해야 하는지)
2. (시나리오 설명)
...
```

**출력 형식 규칙**
- FAIL이 0건이면 "FAIL 항목 상세" 섹션은 생략한다
- 사용자 전달용 시나리오가 없으면 해당 섹션도 생략한다
- 결과표의 "근거" 열은 1~2줄로 간결하게 작성한다

## 엣지 케이스

| 상황 | 처리 방침 |
|------|-----------|
| 체크리스트 없이 코드 파일만 제공된 경우 | 체크리스트를 먼저 요청한다. test-planner를 먼저 실행할 것을 안내한다 |
| 참조할 스킬 .md 명세 파일이 없는 경우 | 사용자에게 명세 파일 경로를 요청한다. 명세 없이는 PASS/FAIL 판정이 불가능하므로 실행을 중단한다 |
| 코드 파일이 존재하지 않는 경우 | 해당 파일을 목록화하여 사용자에게 알리고, 나머지 파일로만 검증을 진행할지 확인한다 |
| 체크 항목이 런타임 동작만 검증하는 경우 (코드 분석 불가) | 해당 항목은 "사용자 전달용 테스트 시나리오"로 분류하고, PASS/FAIL 판정 대상에서 제외한다 |
| 체크 항목이 모호하거나 명세의 어느 부분을 검증하는지 불명확한 경우 | 해당 항목에 "판정 보류 — 명세 구절 불명확"으로 표기하고, 사용자에게 해당 항목의 의미를 확인 요청한다 |
| FAIL 발생 시 task-orchestrator가 없는 경우 | 리포트만 출력하고, 수동 수정 후 `/app-tester`를 재실행할 것을 안내한다 |
| 입력된 코드가 스킬과 무관한 파일(예: 설정 파일, 데이터 파일)인 경우 | 해당 파일은 검증 범위 외로 표기하고, 실제 구현 코드 파일을 추가 요청한다 |
| 체크리스트가 test-planner 형식이 아닌 사용자 직접 작성인 경우 | 항목 파싱을 시도하고, 파싱된 결과를 사용자에게 확인받은 후 진행한다. 형식이 너무 모호하면 각 항목이 어떤 명세를 검증하는지 추가 입력을 요청한다 |
| eval-rubric 결과와 병행하여 사용하는 경우 | 역할 혼동 없이 독립적으로 동작한다. app-tester는 스킬 .md 명세 대조(기능 구현 여부)를, eval-rubric은 결과물의 일반 품질(완성도, 형식, 가독성 등)을 담당한다. 두 결과를 합산하거나 통합하지 않는다 |

## 스킬 연계

- **이 스킬을 호출하는 스킬**: `test-planner` (체크리스트 생성 후 자동 전달), `task-orchestrator` (앱 구현 완료 후 test-planner → app-tester 파이프라인 자동 실행)
- **FAIL 결과를 전달하는 스킬**: `task-orchestrator` (FAIL 항목 수신 시 최대 2회 자동 재작업 트리거)
- **역할 분리**: `eval-rubric`은 결과물의 일반 품질(형식, 완성도, 가독성)을 평가하고, `app-tester`는 오직 스킬 .md 명세와의 일치 여부만 판정한다. 두 스킬은 서로의 영역을 침범하지 않는다
- **단독 사용 가능**: 사용자가 `/app-tester`로 직접 호출 시 모든 단계를 독립적으로 수행한다
