# Verification Before Completion

완료·성공을 주장하기 전에 반드시 실제 검증 명령을 실행하고 결과를 확인한다.

> **플랫폼 중립**: 도구 호출이 없는 순수 방법론이다. Claude Code, Codex 모두 동일하게 적용된다.

---

## 철칙

```
검증 증거 없이 완료를 주장하지 않는다.
```

이 메시지에서 검증 명령을 실행하지 않았다면, 통과했다고 주장할 수 없다.

---

## 게이트 함수 (5단계)

```
완료·성공·통과를 주장하기 전에 반드시:

1. IDENTIFY  : 이 주장을 증명할 명령이 무엇인가?
2. RUN       : 전체 명령을 실행한다 (신선하게, 완전하게)
3. READ      : 전체 출력을 읽고 종료 코드와 실패 수를 확인한다
4. VERIFY    : 출력이 주장을 확인하는가?
               - NO  → 증거와 함께 실제 상태를 명시한다
               - YES → 증거와 함께 주장한다
5. ONLY THEN : 주장한다

어느 단계든 건너뜀 = 검증이 아니라 거짓말
```

---

## 주장별 필요 증거

| 주장 | 필요한 것 | 충분하지 않은 것 |
|------|----------|-----------------|
| 테스트 통과 | 테스트 명령 출력: 0 failures | 이전 실행, "통과할 것 같다" |
| 린터 클린 | 린터 출력: 0 errors | 부분 검사, 추정 |
| 빌드 성공 | 빌드 명령: exit 0 | 린터 통과, 로그 양호 |
| 버그 수정 | 원래 증상 테스트 통과 | 코드 변경, 수정 가정 |
| 요구사항 충족 | 항목별 체크리스트 검증 | 테스트 통과 |
| 에이전트 완료 | VCS diff로 변경 확인 | 에이전트 성공 보고 신뢰 |

---

## 🚩 즉시 멈춰야 할 신호

- "should", "probably", "아마", "~할 것 같다" 사용
- 검증 전 만족감 표현 ("완료!", "통과!", "됐다!" 등)
- 검증 없이 커밋/PR/완료 처리 직전
- 에이전트 성공 보고를 그대로 신뢰
- 부분 검증에 의존
- "이번 한 번만"이라는 생각
- **검증을 실행하지 않고 성공을 암시하는 어떤 표현도**

---

## 합리화 방지

| 변명 | 현실 |
|------|------|
| "이제 작동할 거야" | 검증 명령을 실행하라 |
| "자신 있어" | 자신감 ≠ 증거 |
| "이번 한 번만" | 예외 없음 |
| "린터 통과했어" | 린터 ≠ 컴파일러 |
| "에이전트가 성공했대" | 독립적으로 검증하라 |
| "부분 검사면 충분해" | 부분은 아무것도 증명하지 않는다 |

---

## 패턴 예시

**테스트:**
```
✅ [테스트 명령 실행] [결과: 34/34 통과] → "모든 테스트 통과"
❌ "이제 통과할 것 같아" / "올바르게 보여"
```

**빌드:**
```
✅ [빌드 실행] [exit 0 확인] → "빌드 성공"
❌ "린터 통과했으니 됐겠지"
```

**요구사항:**
```
✅ 스펙 재독 → 체크리스트 작성 → 항목별 검증 → 갭 또는 완료 보고
❌ "테스트 통과했으니 단계 완료"
```

---

## 완료 전 체크리스트

- [ ] 검증 명령을 이 세션에서 실행했는가?
- [ ] 전체 출력을 읽었는가?
- [ ] 출력이 주장을 확인하는가?
- [ ] 다른 테스트가 깨지지 않았는가?

4개 모두 체크 안 됨 → 완료 주장 불가.

---

## 결론

명령을 실행하라. 출력을 읽어라. 그 다음에 주장하라.

예외 없음.
