# test-planner

앱 구현에 사용된 스킬 .md 파일들과 구현 코드를 읽어, 스킬이 약속한 기능들이 실제로 구현됐는지 검증하는 테스트 계획을 수립하는 스킬. 산출물은 코드 분석 기반의 **Claude 테스트 체크리스트**와 실제 환경 기반의 **사용자 테스트 시나리오** 두 가지다.

## 사용법

```
/test-planner
```

실행 후 아래 두 가지를 제공한다:
1. **코드 경로**: 구현된 앱/모듈의 파일 경로 (예: `/home/user/myapp/src/`)
2. **스킬 .md 목록**: 구현에 사용된 스킬 파일 경로들 (예: `rss-fetcher.md`, `web-dev.md`)

`task-orchestrator`가 자동 호출할 때는 위 정보가 자동으로 전달된다.

## 동작 방식

### 1단계: 입력 수집

다음 두 가지 입력을 확인한다.

**A. task-orchestrator 자동 호출 시**
- 코드 경로와 스킬 .md 목록이 자동으로 전달된다 → 바로 2단계로 이동한다

**B. 사용자 직접 호출(`/test-planner`) 시**
- 코드 경로와 스킬 .md 파일 경로를 요청한다
- 코드 경로만 제공되고 스킬 .md가 없는 경우: 사용된 스킬 이름을 물어보거나, `.claude/skills/` 디렉토리를 스캔해 후보 목록을 제시한다
- 스킬 .md만 있고 코드 경로가 없는 경우: 코드 없이는 Claude 테스트 체크리스트를 생성할 수 없으므로, 코드 경로를 요청한다. 코드 없이 사용자 테스트 시나리오만 원하는 경우는 해당 산출물만 제공한다

### 2단계: 스킬 명세 추출

각 스킬 .md 파일을 읽고, **"동작 방식" 섹션**에서 구체적으로 약속된 기능 명세를 추출한다.

추출 기준:
- 수치가 명시된 명세 (예: "5개 프록시", "8초 타임아웃", "최대 3라운드")
- 조건부 동작 (예: "X 실패 시 Y로 폴백", "Z가 없으면 경고")
- 구조적 요건 (예: "4단계로 처리", "체크리스트 + 점수표 두 가지 산출물")
- 사용자에게 약속된 출력 형식

추출 예시 (rss-fetcher.md 기준):
- "2개 프록시를 순서대로 시도한다" → 프록시 배열에 2개 항목이 있는지
- "각 프록시는 5초 타임아웃" → AbortController 5000ms 설정이 있는지
- "RSS 2.0과 Atom 모두 지원" → `item, entry` 셀렉터가 있는지
- "키워드 70% 겹침 기준 중복 필터" → 0.7 threshold 파라미터가 있는지

### 3단계: 구현 코드 분석

추출된 명세 항목을 기준으로 구현 코드를 분석하여, 각 항목이 코드에 실제로 구현되었는지 확인한다.

판정 기준:
- **구현됨**: 명세와 정확히 일치하는 코드가 존재
- **부분 구현**: 기능은 있으나 명세의 수치/조건이 다름 (예: 타임아웃이 5초가 아닌 3초)
- **미구현**: 명세가 약속한 기능이 코드에 없음
- **코드 분석 불가**: 해당 항목이 런타임·환경 의존적이어서 코드만으로 판정 불가 → 사용자 테스트 시나리오로 이관

### 4단계: 테스트 계획 생성

분석 결과를 두 가지 산출물로 정리한다.

#### 산출물 1: Claude 테스트 체크리스트

코드 정적 분석만으로 확인 가능한 항목들. 체크리스트를 생성한 직후 Claude가 직접 코드를 읽어 각 항목의 판정 결과를 채운다. `app-tester` 스킬이 있으면 해당 스킬이 자동 실행하고, 없으면 현재 세션에서 코드를 직접 분석하여 판정한다.

각 항목에 대해:
- 어떤 스킬 명세에서 유래했는지 명시 (출처: 스킬명 + 섹션)
- 코드에서 확인할 구체적인 패턴/값/구조를 명시 (검색어, 함수명, 상수값 등)
- 판정 결과 (구현됨/부분 구현/미구현/분석 불가) 기록

#### 산출물 2: 사용자 테스트 시나리오

브라우저·실행 환경·외부 서비스 등 실제 환경에서만 확인 가능한 항목들.

각 시나리오는 아래 구조로 작성한다:
- **전제 조건**: 테스트 실행 전 준비사항
- **실행 단계**: 순서대로 따라할 수 있는 구체적 행동
- **기대 결과**: 통과 시 나타나야 할 현상

## 출력 형식

### 실제 출력 예시 (rss-fetcher 기반 앱 테스트)

```
## 테스트 계획 — [앱 이름]

참조 스킬: rss-fetcher.md
코드 경로: /home/user/myapp/src/rss.js

---

## Claude 테스트 체크리스트

| # | 출처 명세 | 검증 패턴 | 판정 |
|---|-----------|-----------|------|
| 1 | rss-fetcher: "2개 프록시 순차 시도" | `proxies` 배열 길이 = 2 | ✅ 구현됨 |
| 2 | rss-fetcher: "각 프록시 5초 타임아웃" | `AbortController`, `setTimeout(..., 5000)` | ✅ 구현됨 |
| 3 | rss-fetcher: "RSS 2.0 + Atom 지원" | `querySelectorAll('item, entry')` | ✅ 구현됨 |
| 4 | rss-fetcher: "키워드 70% 기준 중복 필터" | `threshold = 0.7` 또는 `0.7` 상수 | ⚠️ 부분 구현 (0.6으로 설정됨) |
| 5 | rss-fetcher: "parsererror 감지" | `querySelector('parsererror')` | ❌ 미구현 |

**요약**: 5개 항목 중 3개 완전 구현, 1개 부분 구현, 1개 미구현

---

## 사용자 테스트 시나리오

### 시나리오 1: 정상 RSS 수집 확인
**전제 조건**: 앱이 로컬에서 실행 중이고 인터넷 연결이 있음
**실행 단계**:
1. 브라우저에서 앱 URL을 연다
2. 뉴스 피드 영역이 로드될 때까지 기다린다 (최대 10초)
3. 표시된 뉴스 항목의 제목과 날짜를 확인한다
**기대 결과**: 최신 뉴스 항목이 10개 이상 표시되고, 각 항목에 제목·날짜가 있음

### 시나리오 2: CORS 프록시 폴백 동작 확인
**전제 조건**: 브라우저 개발자 도구(Network 탭)를 열어둠
**실행 단계**:
1. 앱 로드 후 Network 탭에서 `allorigins.win` 요청을 확인한다
2. 해당 요청을 브라우저 devtools에서 차단(Block request URL)한다
3. 페이지를 새로고침한다
**기대 결과**: `corsproxy.io`로 재시도하며 뉴스가 정상 표시됨 (또는 "모든 프록시 실패" 오류 메시지가 표시됨)
```

### 미구현 항목 발견 시 추가 출력

```
## ⚠️ 미구현 항목 알림

다음 항목이 스킬 명세에는 약속되어 있으나 코드에서 확인되지 않았습니다:

| 항목 | 출처 | 권장 조치 |
|------|------|-----------|
| parsererror 감지 | rss-fetcher.md 4단계 | XML 파싱 오류 처리 코드 추가 필요 |

task-orchestrator와 연계 중인 경우, 이 항목들을 재작업 목록으로 전달합니다.
```

## 엣지 케이스 처리

| 상황 | 처리 방식 |
|------|-----------|
| 스킬 .md에 "동작 방식" 섹션이 없는 경우 | 스킬 전체 내용에서 동사+목적어 형태의 기능 명세를 직접 추출; 추출 불가 시 사용자에게 알리고 해당 스킬은 건너뜀 |
| 코드가 여러 파일에 분산된 경우 | 제공된 경로 하위 전체 파일을 재귀 탐색; 탐색 범위를 사용자에게 명시 |
| 명세 항목이 동적 로직에 의존해 정적 분석 불가 (예: 런타임 조건 분기) | 해당 항목을 사용자 테스트 시나리오로 이관하고 사유를 명시 |
| 스킬 명세와 코드가 완전히 일치하는 경우 (모든 항목 통과) | "모든 명세 항목이 구현됨" 요약 후 사용자 테스트 시나리오만 제공 |
| 구현 코드가 없거나 경로가 잘못된 경우 | 경로 오류를 안내하고 올바른 경로를 다시 요청; Claude 테스트 체크리스트 생성 불가 명시 |
| 동일한 기능을 여러 스킬이 중복 명세하는 경우 | 중복 명세를 하나의 항목으로 통합하고 출처 스킬을 모두 표기 |
| task-orchestrator에서 미구현 항목 발견 시 | 미구현 항목 목록을 task-orchestrator에 반환하여 eval-rubric 과락 처리 또는 재작업 트리거로 활용 |

## 스킬 연계

- **호출하는 스킬**: `task-orchestrator` (앱 구현 완료 후 자동 호출)
- **결과를 전달하는 스킬**: `app-tester` (Claude 테스트 체크리스트 자동 실행, 해당 스킬이 있는 경우), `task-orchestrator` (미구현 항목 발견 시 재작업 요청)
- **단독 사용 가능**: 사용자가 `/test-planner`로 직접 호출 시 모든 단계를 독립적으로 수행
