# proj_checkers

AI를 활용하는 사용자가 AI를 목적에 맞게, 효과적으로 다루고 있는지 점검하기 위한 스킬 모음 프로젝트.

## 프로젝트 목적

사용자가 AI에게 내리는 지시(프롬프트)의 품질을 높이고, AI의 결과물을 객관적으로 평가할 수 있는 도구를 만든다.

구체적으로:
- 프롬프트가 명확하고 구체적으로 작성되었는지 검토·개선
- AI 결과물이 프롬프트의 요구사항을 얼마나 잘 충족했는지 정량 평가
- 작업 실행 전 갭 스킬을 자동으로 생성하여 스킬 생태계를 점진적으로 확장

---

## AI 에이전트 동작 원칙

### task-orchestrator 자동 발동 규칙

**사용자가 스킬을 별도로 언급하지 않아도** `task-orchestrator`가 자동 발동된다.

**발동 조건** — 다음 중 하나 이상 해당 시:
- 구현·개선·분석·생성·리팩터링 등 실질적 실행 작업을 요청할 때
- 작업 완료에 2개 이상의 스킬이 필요하다고 판단될 때
- "해줘", "만들어", "개선해", "분석해", "작성해", "검토해" 등 실행 지시가 포함될 때

**발동 시 자율 실행 흐름**:
1. 현재 세션에서 접근 가능한 **모든 스킬(85개, 7개 레이어)**을 스캔
2. 3가지 기준으로 적합한 스킬 선정 (아래 참고)
3. 갭 스킬 탐지 → `skill-creator`로 자동 생성 후 본 작업 진행
4. 실행 계획 수립 → **중간 승인 없이** 자율 실행
5. `eval-rubric`으로 품질 검증 (과락 시 최대 2회 자동 재작업)
6. 최종 결과물만 사용자에게 보고 → **최종 승인 1회**

**사용자 개입은 최종 승인 1회만.**

### 스킬 선정 기준 (task-orchestrator 내부 기준)

다음 3가지 중 2개 이상 충족하는 스킬만 실행 계획에 포함한다:

| 기준 | 판단 질문 |
|------|-----------|
| 작업 유형 일치 | 스킬의 목적이 요청 작업과 직접 연관되는가? |
| 파이프라인 적합성 | 이 스킬의 출력이 다음 스킬의 입력으로 자연스럽게 이어지는가? |
| 품질 기여도 | 스킬 없이 직접 처리하는 것보다 결과 품질이 의미 있게 향상되는가? |

### 스킬 레이어 아키텍처 (총 85개)

task-orchestrator는 아래 7개 레이어 전체를 스캔한다.
같은 이름의 스킬이 여러 레이어에 존재하면 **우선순위가 높은 레이어가 자동 적용**된다.

| 우선순위 | 레이어 | 위치 | 스킬 수 |
|:--------:|--------|------|:-------:|
| 1 (최우선) | Project-Local | `.claude/skills/` | 9 |
| 2 | bkit Core v1.6.1 | `~/.claude/plugins/cache/bkit-marketplace/` | 31 |
| 3 | superpowers v5.0.6 | `~/.claude/plugins/cache/superpowers-dev/` | 14 |
| 4 | Anthropic 공식 | `anthropic-skills:*` | 10 |
| 5 | Design | `design:*` | 7 |
| 6 | team-attention-plugins | `clarify:*`, `youtube-digest:*` | 4 |
| 7 | CC 내장 | CC Built-in | 10 |
| | **합계** | | **85** |

현재 레이어 충돌: `skill-creator` — Project-Local이 `anthropic-skills:skill-creator` 오버라이드 (의도된 동작).

```
        ↓

        ↓
  85개 스킬 전체 스캔 → 적합 스킬 선정

  갭 스킬 있음?
   YES ↓           NO ↓
  skill-creator          ↓
  (자동 생성)            ↓
        ↓               ↓
     선정된 스킬 순차 실행 (중간 승인 없음)
        ↓
  eval-rubric 품질 검증
        ↓
  과락 시 최대 2회 자동 재작업
        ↓
  최종 결과물 보고 → 사용자 최종 승인 1회
```

## Project-Local 스킬 목록

### `task-orchestrator` — 작업 실행 팀장 ★ 핵심

구체화된 작업을 받아 **85개 스킬 전체를 스캔**하고, 필요한 스킬을 자율 선택·조율·품질검증까지 완료한다.

- 사용자 별도 스킬 언급 없이 자동 발동
- 갭 스킬 발견 시 `skill-creator`로 자동 생성 후 본 작업 진행
- `eval-rubric` 품질 검증 자동 수행 (과락 시 최대 2회 자동 재작업)
- **사용자 개입: 최종 승인 1회만**

### `prompt-clarify` — 프롬프트 명확화 도우미

사용자 프롬프트를 분석하고 역질문으로 더 명확한 프롬프트를 만든다.

- 최대 3라운드 역질문
- 보유 스킬 추천 및 갭 스킬 탐지
- task-orchestrator 실행 전 선택적으로 사용

### `eval-rubric` — 평가 지표 생성기

완성된 프롬프트 기준으로 AI 결과물을 정량 평가한다.

- 체크리스트 + 점수표 (과락 기준 포함)
- 과락 항목 발생 시 구체적 피드백 제공
- task-orchestrator 마지막 단계에서 자동 실행

### `skill-creator` — 스킬 생성·개선 도구

새 스킬을 처음부터 만들거나 기존 스킬을 개선한다.

- 초안 작성 → 테스트 → 정량 평가 → 반복 개선
- 목표 점수: 95점 이상 달성 시 정식 등록

### `app-tester` — 구현 검증기

스킬 .md 스펙과 실제 구현 코드를 대조하여 PASS/FAIL 판정한다.

### `test-planner` — 테스트 케이스 생성기

구현 전 테스트 체크리스트를 생성한다. `app-tester`와 연계.

### `rss-fetcher` — RSS 파서

브라우저 환경에서 RSS/Atom 피드를 파싱하고 중복을 제거한다.

- 5단계 CORS 프록시 폴백
- 키워드 기반 중복 제거 (기본 임계값 0.7)

### `gemini-client` — Gemini API 클라이언트

브라우저에서 Google Gemini API를 호출한다.

- API 키 localStorage 저장 (소스코드에 절대 포함 금지)
- 모델 폴백: gemini-2.5-flash → 2.0-flash-001 → 1.5-flash

### `web-dev` — 웹앱 개발

정적 웹 애플리케이션 개발 가이드. 바닐라 JS 기반.

## 개발 가이드라인

### 스킬 추가 원칙

- 각 스킬은 `.claude/skills/` 에 독립된 `.md` 파일로 작성
- 스킬 파일 구조:
  1. YAML 프론트매터 (`name`, `description` 필수)
  2. 스킬 이름 및 한 줄 설명
  3. 사용법 (슬래시 커맨드)
  4. 동작 방식 (단계별)
  5. 출력 형식 예시
- 하나의 스킬은 하나의 명확한 목적만 수행한다
- 신규 스킬은 `skill-creator`로 초안 작성 후 95점 이상 달성 시 정식 등록

### 스킬 품질 기준

- YAML 프론트매터의 `description`은 트리거 조건을 구체적이고 능동적으로 명시
- 사용자 입력이 모호할 경우 스킬 스스로 명확화 질문을 해야 한다
- 출력은 사용자가 즉시 활용할 수 있는 형식이어야 한다
- 정량 평가가 포함된 스킬은 반드시 수치 기준(과락선 등)을 명시해야 한다
- 엣지 케이스(모호한 입력, 오류, 예외 상황)에 대한 처리 방침을 명시해야 한다

### 갭 스킬 처리 원칙

- `prompt-clarify` 실행 시 갭 스킬이 탐지되면 결과물에 명시한다
- `task-orchestrator` 실행 시 갭 스킬이 있으면 `skill-creator`로 먼저 생성한 뒤 본 작업을 진행한다
- 생성된 갭 스킬은 95점 기준으로 iteration하여 품질을 확보한다

## 향후 추가 예정 스킬 (아이디어)

- 프롬프트 유형 분류기: 주어진 프롬프트가 어떤 유형(지시형, 질문형, 역할부여형 등)인지 분류
- 프롬프트 비교 평가기: 두 버전의 프롬프트 중 어느 쪽이 더 효과적인지 비교 분석
- AI 편향 점검기: AI 결과물에 특정 편향이 포함되어 있는지 점검
